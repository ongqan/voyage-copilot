## 本次产品优化的关键结论

1. MVP的验证闭环收敛为：**行程导入 → 确认 → 权益判断 → 推荐 → 解释 → 二次确认 → 模拟订单 → 客服接管**。
2. “异常履约”存在范围冲突。MVP只做模拟异常、影响识别、用户方案和工单闭环；实时航变接入与自动改订放到V1.1。
3. 大模型不负责资格、价格、点数、库存和订单状态的最终判断；这些结果必须来自结构化数据和规则引擎。
4. 所有订单写操作必须由服务端校验一次性确认令牌，不能依赖Prompt约束。
5. MVP采用模块化单体、有限状态AI工作流和可解释加权推荐，暂不建设微服务、开放式Agent或机器学习排序。

## 开发前需要业务方确认的事项

以下事项已经给出推荐默认值，可以先开工，但必须在第2周结束前确认：

| 决策 | 推荐默认值 | 最晚确认 | 影响 |
|---|---|---|---|
| 首发区域 | 中国大陆5个示范机场，不含高铁 | W1 | 数据、规则、原型文案 |
| 首发服务 | 贵宾室、餐饮、快速安检 | W1 | 主流程和测试集 |
| 接送机 | 仅保留演示数据，不进P0 | W2 | 时间线复杂度 |
| 航班数据 | 完全模拟，可手工注入航变 | W2 | 外部依赖与成本 |
| 登录方式 | Demo账号+预置会员身份 | W2 | 身份、授权和测试 |
| 通知渠道 | 仅站内消息 | W2 | 隐私、供应商和排期 |
| 语言 | 中文为上线门禁，英文为P1 | W2 | 内容、UI、评测量 |
| 模型与部署区 | 通过ADR选择，禁止业务层绑定供应商 | W2 | 合规、成本、延迟 |
| 客服SLA | 普通4小时、重要1小时、紧急15分钟 | W2 | 队列与工单 |
| 试点方式 | 内部+受邀用户，所有交易均为模拟 | W2 | 实验结论边界 |

## 评审顺序

1. 产品范围评审：PRD、体验流程、规则规格；
2. 技术评审：系统、数据/API、AI设计；
3. 交付评审：测试计划、路线图、Backlog；
4. 原型走查：逐状态检查正常、异常、无数据、低置信度和无权限场景；
5. 启动评审：Owner、容量、环境、指标口径和首个迭代冻结。

## Ready for Development 标准

一个Story只有同时满足以下条件才可进入研发：

- 用户价值、范围和非范围明确；
- 主流程、异常流程、权限和页面状态明确；
- 依赖接口、数据字段、规则版本和错误码明确；
- 已定义埋点、审计要求和验收用例；
- 涉及AI时已定义证据、工具、置信度与降级方式；
- 涉及写操作时已定义确认、幂等和回滚机制；
- 设计稿或交互说明已通过产品、设计、研发、QA评审。
